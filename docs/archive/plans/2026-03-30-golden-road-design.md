# 黄金之路设计方案

日期：2026-03-30

## 1. 模块定位

“黄金之路”不是一个技能展示组件，而是 NexusNote 的长期成长导航系统。

它的核心职责不是回答“你学过哪些技能”，而是回答以下问题：

- 我现在正在朝哪个职业方向成长
- 我已经具备哪些关键能力，证据是什么
- 当前课程和章节，正在推进哪些技能节点
- 距离目标职业路线还缺哪些能力
- 下一步最值得学习的技能是什么
- 如果继续学下去，我还有哪些可能转向的新路线

因此，“黄金之路”应该成为课程系统、学习助手、笔记系统、沉淀系统和个人档案之间的上层抽象。

它是一个用户视角的“成长操作系统”，而不是内部视角的“技能表 + 关系表”。

## 2. 为什么现有技能模块不够

现有实现的问题不是 UI 不够好看，而是产品抽象层级过低：

- 只有离散技能节点，没有职业路线
- 只有关系，没有明确的成长顺序和先修结构
- 课程与章节没有显式映射到技能
- 技能进度缺少“证据驱动”的可信来源
- 用户看不出当前学习对长期目标的作用
- 推荐只是“下一个相关技能”，不是“下一个最值得走的成长路径”

用户实际需要的不是图，而是“成长判断系统”。

所以“黄金之路”必须从“技能图谱”升级为：

`职业路线 -> 能力域 -> 技能节点 -> 学习证据 -> 下一步行动`

## 3. 北极星体验

用户进入“黄金之路”后，应该在 10 秒内看到三件事：

1. 我当前最接近的职业路线是什么
2. 我在这条路线上的进度到哪里了
3. 我接下来该补哪几个关键技能

用户点击任意职业路线后，应该立即理解：

- 这条路线需要哪些核心能力
- 我已掌握哪些
- 我正在通过哪些课程和章节推进这些能力
- 还缺哪些关键节点
- 下一门课或下一章应该学什么

这个模块最终要让用户产生一种非常明确的感受：

“我不是在散着学内容，我是在沿着一条职业成长路径前进。”

## 4. 产品结构

黄金之路建议拆成 4 个主视图：

### 4.1 路线总览

目标：让用户快速看到自己的主路线和次路线。

内容：

- 当前主路线卡片
- 当前主路线总进度
- 已掌握技能数 / 学习中技能数 / 待解锁技能数
- 当前课程正在点亮的技能
- 下一步推荐技能
- 可延展的其他职业路线

### 4.2 路线地图

目标：以“网状、可成长、可推进”的形式展示路线结构。

设计方向：

- 不做思维导图式随意散点
- 采用“命途 / 星盘 / 经络”式网络结构
- 节点按能力域聚类
- 聚类之间通过依赖路径连接
- 当前路线高亮，其他路线弱化

节点状态：

- 已掌握：强高亮、实体发光
- 学习中：半高亮、动态脉冲
- 可开始：实边框、待点亮
- 未解锁：低透明、锁定态

### 4.3 职业路线详情

目标：让用户对单条路线形成完整认知。

内容：

- 路线说明：适合什么人、典型目标岗位、典型产出
- 路线所需能力域
- 各能力域下的技能节点
- 用户的已掌握/学习中/缺失能力
- 技能缺口优先级
- 推荐课程与章节

### 4.4 技能节点详情

目标：把技能从“抽象标签”变成“可行动对象”。

内容：

- 技能定义
- 隶属能力域
- 前置技能 / 延展技能 / 替代技能
- 与之相关的课程与章节
- 用户当前证据
- AI 评估掌握度
- 下一步学习建议

## 5. 领域模型

黄金之路至少需要 5 层结构：

### 5.1 动态成长主线 Path Projection

“路线”不应该是平台预置的死分类，而应该是用户在统一能力图谱上的动态投影。

也就是说：

- 平台底层维护的是开放能力图谱
- 用户目标、课程、章节、笔记、沉淀会持续提供证据
- 系统据此动态生成“当前主线”“潜在线”“下一跳方向”

因此，路线描述的不是平台固定菜单，而是用户当前最值得推进的能力组合。

### 5.2 能力域 Capability Domain

例如：

- 产品判断
- 技术实现
- 数据能力
- AI 应用设计
- 商业与增长
- 项目化交付

能力域是路线中的中层骨架。

### 5.3 技能节点 Skill Node

例如：

- 用户研究
- Prompt Engineering
- React
- SQL
- Python 数据处理
- LLM 应用架构
- 实验设计

技能节点必须足够具体，能够映射到学习内容和成长证据。

### 5.4 学习证据 Evidence

技能进度不能只看课程完成率，必须来自证据。

证据来源包括：

- 相关课程是否存在
- 相关章节是否完成
- 学习对话中是否深入涉及该技能
- 是否有高亮
- 是否有笔记
- 是否有沉淀
- 是否有练习 / 项目产出

### 5.5 路线关系 Relationship

关系包括：

- prerequisite：前置
- unlocks：解锁
- strengthens：增强
- adjacent：邻接路线
- alternative：替代路线

## 6. 关键产品机制

### 6.1 技能状态不再只用完成度表示

建议把每个技能拆成两种值：

- 覆盖度：你接触了多少相关内容
- 掌握度：你真正表现出了多少能力

这样就能区分：

- 学过但不会
- 正在练但还不稳
- 已经掌握

### 6.2 路线进度是加权的

职业路线进度不应该由技能数量平均计算，而应由路线权重决定。

例如一个用户当前主线偏向“AI 产品构建”，那么：

- 产品判断
- AI 应用设计
- 数据实验能力

这些核心节点的权重要明显高于外围技能。

因此需要对每个技能节点设置：

- importance：该路线中的重要性
- tier：foundation / core / advanced

### 6.3 下一步推荐必须是“路线内最优决策”

推荐逻辑不应只是“相关技能”。

应优先推荐：

- 已满足先修条件
- 对当前主路线贡献最大
- 与当前正在学课程高度衔接
- 能显著降低路线缺口的技能

## 7. 开放式路径设计建议

V1 不应把职业路线做成平台固定死菜单，而应采用：

- 统一能力域
- 统一技能节点
- 用户意图抽取
- 动态路径投影

### 7.1 平台固定的是“能力图谱”，不是“职业列表”

平台可以维护稳定的能力域，例如：

- 产品判断
- 前端体验
- 系统与数据
- AI 应用架构
- 数据分析
- 自动化杠杆
- 增长与商业
- 交付与运维

这些是底层能力骨架，可以长期存在。

### 7.2 用户看到的是“当前主线”，不是预设菜单

系统根据用户目标与学习证据，生成类似这样的主线：

- AI 产品主线
- AI 构建主线
- 数据洞察主线
- 独立构建主线
- 或更个性化的组合主线

这些名称和结构可以由系统动态合成，而不是死写在产品里。

### 7.3 可以有内部先验，但不能暴露成产品真相

系统内部可以维护若干 archetype 或 prior，帮助做初步归因和推荐。

但它们只能作为：

- 推断起点
- 解释模板
- 推荐排序参考

不能成为用户界面里“平台只有这几条路”的表达。

## 8. 与课程系统的关系

黄金之路必须和课程系统做双向绑定。

### 8.1 课程 -> 技能

每门课程都应回答：

- 这门课覆盖哪些技能节点
- 这门课更偏向哪条动态成长主线
- 学完这门课后会推进哪些能力域

### 8.2 章节 -> 技能

每个章节都应回答：

- 本章在训练哪些技能
- 这些技能在当前路线中处于什么位置
- 学完本章后会解锁什么

### 8.3 技能 -> 课程 / 章节

每个技能节点都应反向展示：

- 哪些课程覆盖它
- 哪些章节最直接训练它
- 用户已经通过哪些课程获得该技能证据

## 9. 与笔记和沉淀系统的关系

黄金之路不应只吃课程进度，还要吃知识工作台的数据。

因为高亮、笔记、沉淀，才是“理解发生了”的证据。

建议：

- 高亮：提供覆盖度证据
- 笔记：提供理解度证据
- 沉淀：提供结构化掌握证据

后续可以通过 AI 对沉淀内容进行技能归因：

“这条沉淀主要体现了实验设计和数据解释能力”

这样技能就不再只是课程内产物，而是用户整个学习系统的综合结果。

## 10. UI / 视觉方向

风格上不要做普通知识图谱，也不要做 B 端系统拓扑图。

更适合的方向是：

- 带神话感和命途感的成长网络
- 深色中性底 + 金色/琥珀色高亮
- 有层级、能量、路径感
- 强调“被点亮”和“待解锁”

视觉关键词：

- constellation
- path of mastery
- mythic progression
- elegant strategic interface

但要避免炫技：

- 信息可读性必须高于视觉表演
- 节点必须可点、可筛选、可理解
- 右侧详情面板必须稳定

## 11. 页面信息架构

V1 推荐新增独立页面：

- `/career-trees`

页面结构：

### Hero

- 当前主线名称
- 主线一句话描述
- 当前总进度
- 已掌握 / 学习中 / 缺失统计

### Path Navigator

- 当前主线
- 潜在线
- 下一跳方向

### Road Network

- 按能力域分组的技能网络
- 连接关系
- 当前激活路线高亮

### Route Gaps

- 当前主线还缺的关键技能
- 哪些是硬缺口
- 哪些是可以后补的外围技能

### Linked Learning

- 正在学习的课程与章节
- 每门课映射到哪些技能

### Future Routes

- 你还可以延展到哪些潜在线
- 各潜在线当前匹配度

## 12. 数据设计建议

最终目标模型建议新增：

- `capability_domains`
- `capability_skills`
- `capability_skill_edges`
- `user_growth_profiles`
- `user_path_projections`
- `user_skill_evidence`

但 V1 不建议一上来就大迁移。

### V1 方案

使用“服务端聚合视图”先做：

- 静态能力域与技能 ontology
- 复用现有 `skills`
- 复用现有 `user_skill_mastery`
- 复用 `courses` / `course_progress`
- 从课程标题、章节标题、技能表中推断映射
- 在服务端生成动态 path projection

V1 的进度判定建议先只使用课程系统信号：

- 课程覆盖哪些技能
- 章节覆盖哪些技能
- 用户完成了哪些章节

高亮、笔记、沉淀先只作为旁路证据展示，不进入主进度分数。

这样可以先验证产品价值，再决定是否做正式数据建模。

## 13. 推荐算法建议

V1 推荐算法可以先用规则驱动：

### 输入

- 当前主路线
- 用户已掌握节点
- 学习中节点
- 缺失节点
- 当前课程上下文

### 输出

- next best skills
- route gap priorities
- adjacent routes

### 规则优先级

1. 已满足先修条件的核心技能
2. 被当前课程强覆盖的技能
3. 对路线进度贡献最大的技能
4. 与未来可延展路线共享的技能

V2 再考虑引入 AI 对成长路径进行动态排序。

## 14. 推荐的 V1 落地范围

为了尽快让模块可用，V1 只做以下能力：

- 动态主线生成
- 主线总览页
- 主线详情页
- 技能节点状态展示
- 当前课程 / 章节与技能的映射
- 缺口分析
- 下一步建议

V1 暂时不做：

- 用户自由编辑底层图谱
- 完整自由拖拽图谱编辑
- 实时每请求重新生成整条主线
- 多人协作成长图

## 15. 开发建议

建议按下面顺序实施：

### Phase 1: 服务端聚合与页面骨架

- 建立能力 ontology
- 建立技能节点聚合逻辑
- 建立 path projection 逻辑
- 新增 `/career-trees`

### Phase 2: Profile 入口替换

- 在 profile 中用“黄金之路预览”替代旧 SkillGraph
- 增加跳转入口

### Phase 3: 课程与章节映射增强

- 章节级技能映射可视化
- 路线缺口与课程推荐联动

### Phase 4: 正式数据模型升级

- 将用户主线、证据、路径聚焦持久化

## 16. 成功标准

黄金之路做对之后，用户应该能够：

- 清楚知道自己在走哪条成长路线
- 知道当前课程为什么重要
- 知道学完当前章节后能力发生了什么变化
- 知道自己还缺什么
- 知道下一步该学什么

如果用户看完页面后仍然只觉得“这是个技能展示图”，那这个模块就还没做对。

## 17. 推荐命名

模块名称：

- 黄金之路

页面标题：

- 黄金之路
- 你的成长路线

副标题建议：

- 把课程、章节、技能与职业目标连接成一条真正可推进的成长路径。
