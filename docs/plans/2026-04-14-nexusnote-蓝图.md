# NexusNote 蓝图

## 一句话定义

NexusNote 不是课程平台，也不是笔记工具。

它的目标是：

> 一个以学习成长为中心的 AI 知识操作系统。

这句话的真正含义是：

- 课程是入口，不是终点
- 对话是运行时调度层，不是独立产品
- 笔记是知识沉淀层，不是孤立文档
- 职业树是成长导航层，不是展示玩具
- 页面都只是同一套用户知识流的不同投影

## 我们为什么要做这个产品

大多数学习产品有两个根本问题：

1. 只会交付内容，不会保留成长
2. 只会保存笔记，不会重组学习

也就是说，用户可以：

- 听完课程
- 记下笔记
- 问几个问题

但系统并不会真正理解：

- 用户现在到底长出了哪些能力
- 哪些知识已经沉淀成稳定理解
- 下一步最值得补什么
- 之前的学习如何影响之后的课程和路径

NexusNote 要解决的，不是“再做一个更聪明的课程平台”，而是：

> 把目标、课程、学习行为、对话、笔记、沉淀、成长方向，连成一套持续演化的系统。

如果用系统语言把它再收紧，这句话等价于：

> 把所有学习相关行为先沉淀为 `evidence event`，再由系统把这些事件组织成证据、洞察、成长图和多视图投影。

## 对外定位

对用户，我们不讲“液态知识”这种内部术语。

最合适的对外定位是：

> AI 学习知识操作系统

可以展开成三种表达：

1. 把学习目标变成课程，把课程变成成长路径的 AI 学习知识操作系统
2. 把课程、对话、笔记和成长路径连成一体的 AI 学习系统
3. 从目标到课程、从课程到沉淀、从沉淀到成长图谱的 AI 学习操作系统

不建议对外讲：

- 液态知识平台
- 图谱学习引擎
- ontology-first 系统

这些是内部架构语言，不是用户价值语言。

## 对内定义

对内，我们可以用一个更准确的架构定义：

> NexusNote 的底层目标是一个液态知识系统。

“液态知识”在这里不是潮词，而是工程约束。

它意味着知识必须满足五个条件：

1. **跨容器流动**  
   课程、章节、对话、笔记、沉淀、职业树之间不是孤岛。

2. **身份连续**  
   用户看到的名字和结构可以变化，但系统必须知道“这还是同一条能力分支”。

3. **多视图投影**  
   课程页、Editor、对话页、职业树页、Profile 都不是本体，而是投影。

4. **双向回流**  
   课程影响职业树，职业树反过来影响学习焦点；笔记和对话也应进入这条回流链。

5. **持续重组**  
   系统不能只记录知识，还要持续重组用户当前最值得看的结构。

## Evidence Event 是什么

在 NexusNote 里，最底层不应该把“知识”直接理解成：

- 一门课程
- 一条消息
- 一篇笔记
- 一条树枝

这些都只是容器或投影。

真正更底层的统一对象应该是：

> `evidence event`

它表示：

- 某个知识信号发生了
- 它来自某个来源
- 它发生在某个时间
- 它带有上下文、置信度和引用关系
- 它可以被后续系统吸收进 evidence / insight / growth graph

### Evidence Event 的来源至少包括

- 课程大纲抽取
- 课程学习进度变化
- 高亮 / 标注
- 笔记创建和修改
- 学习对话
- capture 沉淀
- 用户选择主树和偏好行为

### 为什么必须引入它

如果没有 `evidence event`，系统最后还是会变成：

- 课程是一套真相
- 笔记是一套真相
- 对话是一套真相
- 职业树是一套真相

只有先把来源统一成事件流，系统才真正有机会做到：

- 统一
- 可追踪
- 可回流
- 可重组
- 可投影

## 产品结构

最终产品应该分成六层，不是一堆功能页并列。

### Layer 1：目标层 Goal

承载用户目标进入系统。

页面：

- 首页
- 课程访谈

职责：

- 收目标
- 收背景
- 收应用场景
- 决定是否足够生成课程蓝图

### Layer 2：蓝图层 Blueprint

把目标转成结构化学习蓝图。

页面 / 交互：

- 访谈大纲预览
- 课程蓝图生成

职责：

- 给系统提供第一批结构化学习骨架
- 为后续学习和职业树提供初始证据源

### Layer 3：学习层 Learning

这是产品的主工作流层。

页面：

- 学习页
- 小节生成
- 学习对话

职责：

- 让用户真正推进内容
- 产生学习行为和进度
- 产生可被系统吸收的 `evidence event`

### Layer 4：知识层 Knowledge

这里不是“写文档”，而是把学习结果变成可复用知识。

页面：

- 高亮
- 笔记
- 沉淀
- Editor 工作台

职责：

- 把原始学习内容转成 evidence、capture 与可重组知识单元
- 让用户可以重看、重组、再理解

### Layer 5：成长层 Growth

这里负责“成长方向”和“下一步”。

页面：

- 职业树
- 成长主线
- 当前焦点 / 下一步

职责：

- 维护隐藏能力图的连续性
- 给用户提供候选成长方向
- 给系统提供学习导航约束

### Layer 6：总览层 Meta

这里只做概览，不做系统本体。

页面：

- Profile
- Settings
- Insights

职责：

- 总结状态
- 配置偏好
- 提供洞察入口

## 四个核心模块的真正角色

### 1. 课程

课程不是产品中心，而是学习入口和训练场。

它负责：

- 把目标转成结构化学习蓝图
- 提供最稳定的初始知识来源

### 2. 对话

对话不是客服，也不是普通聊天。

它负责：

- 解释
- 调整
- 澄清
- 追问
- 捕捉运行时知识信号

所以对话的本质是：

> 运行时操作层

### 3. Editor / 笔记工作台

它不该只是笔记列表页，而应该成为：

> 知识沉淀与重组工作台

它最终应该承接：

- 高亮
- 课程笔记
- 对话 capture
- insight 组织

### 4. 职业树

职业树不该喧宾夺主，但也不是可有可无。

它的真正角色是：

> 成长导航层

它负责回答：

- 我现在在长什么
- 哪些方向正在变强
- 哪条路最值得继续走

## 信息架构原则

### 原则 1

首页不是控制台，也不是知识中心。  
首页只负责启动。

### 原则 2

学习页才是主现场。  
大部分真实用户行为应围绕 Learn 发生。

### 原则 3

Editor 不是附属功能。  
它是知识层最重要的工作台。

### 原则 4

职业树负责导航，不负责承载全部知识。  
它不能替代课程，也不能替代工作台。

### 原则 5

Profile 只做总览，不做控制台。  
不让所有功能模块在这里争主角。

## 数据与系统观

最终系统不该按页面功能建模，而要按知识流层次建模。

内部长期目标应形成六层数据结构：

1. **来源层**
   - 课程
   - 对话
   - 笔记
   - 标注

2. **事件层**
   - 统一 `evidence event`

3. **证据层**
   - 从 event 聚合出的稳定 evidence

4. **洞察层**
   - 主题
   - 缺口
   - 强项
   - 推荐理由

5. **成长层**
   - 用户隐藏能力图
   - 能力边

6. **投影层**
   - 职业树快照
   - 焦点快照
   - Profile 快照

## 最终产品终态

如果蓝图真正完成，NexusNote 应该表现成下面这样：

1. 用户说出目标
2. 系统生成课程蓝图
3. 用户学习内容并留下进度、对话、笔记和高亮
4. 这些行为被抽成统一证据
5. 证据更新洞察层和隐藏能力图
6. 隐藏能力图驱动候选职业树和下一步焦点
7. 焦点继续影响课程组织、对话引导和知识工作台排序

做到这里，产品才不是几个功能拼在一起，而是一套真正的知识操作系统。

## 成功标准

当下面这些都成立时，我才会说蓝图真正实现了：

1. 同一批用户知识可以跨课程、对话、笔记、职业树流动
2. 用户成长分支在多次再生成后仍连续
3. 课程、笔记、对话、职业树都不再是孤立真相源
4. 系统不只是记录学习，还会反过来重组学习
5. 页面只是视图，底层知识结构才是本体

## 最终定位

如果要用一句完整的话总结：

> NexusNote 是一个以课程为入口、以液态知识为底座、以成长图谱为导航、以行动推荐为输出的 AI 学习知识操作系统。
